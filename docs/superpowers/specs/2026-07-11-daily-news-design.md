# 每日新闻日刊设计

**日期：** 2026-07-11
**状态：** 已确认
**参考项目：** [ruanyf/weekly](https://github.com/ruanyf/weekly)

## 1. 背景

本项目从空仓库开始，目标是建设一个无需付费服务、每天自动发布的新闻日刊。项目借鉴
`ruanyf/weekly` 的“一期一份 Markdown、首页归档、稳定历史链接”模式，但将人工周刊改造成
适合长期运行的自动化日刊。

首版不使用 AI、翻译服务或需要个人密钥的新闻 API。内容来自公开 RSS/Atom 和公开 API，
只展示标题、来源、时间、短摘要与原文链接。

## 2. 目标

1. 每天北京时间 09:00 自动生成前一天的日刊。
2. 当前一天新闻不足时，可从此前两天补充，并明确标注“近三日补充”。
3. 使用“一天一份 Markdown”保存内容，确保仓库本身可阅读、可编辑、可追溯。
4. 自动生成最新一期、月度归档、来源说明和 RSS，并部署到 GitHub Pages。
5. 新闻源通过配置文件维护，增删来源不要求修改程序代码。
6. 对来源失败、重复内容、漏跑和重复运行有明确的降级与恢复机制。
7. 整套流程只依赖 GitHub Actions 自动提供的权限，不要求用户配置个人密钥。

## 3. 非目标

首版不包含以下功能：

- AI 摘要、自动翻译或观点评论。
- 新闻全文、付费墙内容或媒体图片转载。
- 登录、评论、收藏、个性化推荐或后台管理系统。
- 数据库和常驻服务器。
- 严格到秒的 09:00 发布 SLA；GitHub Actions 可能因平台负载延迟。
- 自动判断新闻事实真伪或政治立场。
- 全站搜索。首版使用月度归档，后续可基于生成的索引增加纯前端搜索。

## 4. 方案选择

采用“Python 采集与生成 + GitHub Actions 调度 + GitHub Pages 托管”的方案。

未采用 Jekyll 作为核心生成器，原因是它会同时引入 Python 与 Ruby 两套工具链。
未采用仅更新 README 的方案，原因是日刊每年会产生约 365 期，
长期归档和阅读体验会快速下降。

## 5. 内容与页面形态

### 5.1 日刊文件

每期使用日期作为唯一标识，不另外维护连续期号：

```text
content/2026/07/2026-07-10.md
```

文件包含 YAML 元数据和 Markdown 正文。元数据至少包括：

- `date`：本期新闻日期，使用北京时间。
- `generated_at`：生成时间，使用带时区的 ISO 8601 时间。
- `article_count`：最终新闻数量。
- `source_count`：最终来源数量。
- `fallback_count`：来自此前两天的补充数量。
- `categories`：本期实际出现的栏目。

标题格式为“每日新闻 · YYYY-MM-DD”。正文使用固定栏目顺序：

1. 国内与国际
2. 商业与经济
3. 科技与互联网
4. 科学与健康
5. 文化与生活
6. 体育

没有内容的栏目不输出。每条新闻包含：

- 原文标题与链接。
- 来源名称。
- 原始发布时间。
- 最长约 180 个字符的纯文本摘要；没有可安全使用的摘要时只展示标题。
- “近三日补充”标记，仅用于不属于目标日期的补充内容。

### 5.2 网站页面

静态网站至少生成：

- 首页：最新一期、栏目导航和最近十期入口。
- 单期页：完整日刊内容。
- 月度归档页：按月份列出所有期刊。
- 总归档页：按年份和月份导航。
- 来源页：列出已启用来源、栏目、语言和来源类型。
- 关于页：说明生成规则、版权边界，并链接到仓库的 GitHub Issues 作为反馈入口。
- `rss.xml`：提供最近 30 期的订阅入口。

页面采用阅读优先、移动端适配的轻量样式，不加载第三方脚本、字体或跟踪服务。

### 5.3 README

README 说明项目定位、网站地址、本地运行方式和自动化配置，并通过受控标记区自动维护
最近十期链接。完整历史归档留在网站中，避免 README 无限增长。

## 6. 系统架构

系统按职责拆分为以下组件：

1. **配置加载器**：读取来源、栏目、配额和全局抓取配置。
2. **来源抓取器**：分别处理 RSS/Atom 和公开 JSON API。
3. **标准化器**：统一标题、链接、摘要、时间、语言和来源字段。
4. **去重器**：处理同链接、跟踪参数、重复标题和历史重复。
5. **筛选器**：按时间、来源优先级、栏目配额和来源多样性选择内容。
6. **日刊渲染器**：生成日期 Markdown 和 README 最近期刊区域。
7. **站点生成器**：从内容目录生成 HTML、归档页和 RSS。
8. **命令行入口**：提供生成、构建和校验命令，供本地与 GitHub Actions 共用。

建议的仓库结构：

```text
.github/workflows/
  ci.yml
  publish.yml
config/
  sources.toml
content/
  YYYY/MM/YYYY-MM-DD.md
src/day_news/
  cli.py
  config.py
  models.py
  normalize.py
  dedupe.py
  select.py
  render.py
  site.py
  fetchers/
    rss.py
    hacker_news.py
templates/
assets/
tests/
  fixtures/
docs/superpowers/
pyproject.toml
README.md
```

构建产物写入忽略版本控制的 `dist/`。版本库提交内容 Markdown 和源代码，不提交重复的 HTML
构建产物；GitHub Pages 由工作流部署 `dist/`。

## 7. 数据模型

标准化后的新闻条目至少包含：

- `id`：来源 ID、GUID 或规范链接生成的稳定标识。
- `title`：清洗后的原文标题。
- `url`：原始访问链接。
- `canonical_url`：移除跟踪参数、片段和无意义尾斜杠后的链接。
- `source_id` 与 `source_name`。
- `category`：映射后的固定栏目。
- `published_at`：带时区的原始发布时间。
- `fetched_at`：抓取时间。
- `summary`：去除 HTML 后的短摘要。
- `language`：来源配置声明的主要语言。
- `is_fallback`：是否来自目标日期之前的两天。
- `rank_key`：用于确定性排序的键。

来源配置至少包含：

- 唯一 ID、展示名称、来源类型和 URL。
- 所属栏目和主要语言。
- 优先级、是否启用和单来源最大入选数。

## 8. 新闻来源策略

官方 RSS/Atom 是主要来源，Hacker News 官方 API 作为科技与互联网栏目的补充。首版不把未正式
承诺稳定性的聚合 RSS 作为核心依赖。

来源配置遵循以下规则：

- 每个核心栏目配置至少两个相互独立的来源。
- 同一媒体的不同栏目仍计为同一来源，避免单一媒体占据整期。
- 优先使用来源方公开提供的 RSS/Atom 或正式公开 API。
- 不抓取需要登录、绕过反爬、绕过付费墙或禁止自动访问的页面。
- 来源失效时可以仅修改 `config/sources.toml` 完成替换。

国际来源保留原文标题，不在无翻译服务的情况下伪造中文翻译。
中文与其他语言内容可以共存，来源页明确标注主要语言。

## 9. 每日生成流程

### 9.1 时间窗口

定时任务运行时，以 `Asia/Shanghai` 计算目标日期：

```text
目标日期 = 当前北京时间日期 - 1 天
```

抓取窗口覆盖目标日期及此前两天，共 72 小时。
优先选择目标日期 00:00 至 24:00 的内容；
只有目标日期内容无法满足数量或栏目覆盖要求时，才从此前两天补充。

缺少 `published` 和 `updated` 时间的条目不进入日刊，避免将抓取时间误当成新闻发布时间。

### 9.2 规范化与去重

链接规范化会移除常见跟踪参数、URL 片段和多余尾斜杠。去重顺序为：

1. 来源 GUID 或稳定 ID。
2. 规范链接完全相同。
3. 标题经过 Unicode 规范化、空白和标点清洗后完全相同。
4. 规范化标题的 `SequenceMatcher` 相似度达到 0.92 时视为同一条候选新闻。

程序同时读取最近 30 天内容中的规范链接和标题，避免跨期重复。
不同媒体报道同一事件时，首版只保留排序最靠前的一条，暂不创建多来源事件聚类。

### 9.3 筛选规则

筛选过程必须确定性运行，相同输入产生相同输出。默认约束为：

- 每期目标 24 条，最低 12 条，最高 30 条。
- 至少覆盖 4 个栏目和 5 个独立来源。
- 每个来源默认最多入选 3 条，来源配置可设置更低上限。
- 排序键依次为：是否属于目标日期、来源优先级、发布时间、来源内位置、规范链接。
- 目标日期无法达到发布门槛时，再使用此前两天的候选条目补齐。

如果最终未达到最低 12 条、4 个栏目或 5 个来源中的任一门槛，
本次生成失败且不提交期刊。

## 10. 定时与发布

`publish.yml` 同时处理定时生成和 Pages 部署：

- 主调度：UTC 01:00，即北京时间 09:00。
- 备用调度：UTC 01:20，即北京时间 09:20。
- 手动调度：支持指定目标日期；仅手动运行允许使用强制重建参数。
- 主调度成功生成并提交后，备用调度发现日期文件已存在便直接成功退出。
- 使用并发组防止同一仓库的两个发布任务同时写入。

GitHub Actions 定时任务可能因平台负载延迟，因此 09:00 是目标启动时间而非严格 SLA。
每日流程具有幂等性；同一天重复运行不会增加第二份期刊或产生无意义提交。

发布流程为：

1. 检出默认分支并安装锁定的 Python 依赖。
2. 运行测试和内容配置校验。
3. 定时或手动事件生成日刊；普通 push 事件跳过抓取。
4. 内容发生变化时，以 GitHub Actions 机器人身份提交并推送。
5. 在当前工作区构建 `dist/`。
6. 上传 Pages 构建产物并部署 GitHub Pages。

生成提交使用 `GITHUB_TOKEN` 后通常不会触发新的 push 工作流，
因此定时任务必须在同一次运行中继续完成 Pages 部署，
不能依赖机器人提交再次触发另一个工作流。

## 11. 失败处理与可观测性

- 每个来源单次请求超时为 10 秒；首次失败后最多再重试两次。
- 单个来源失败只记录警告，不立即终止全局流程。
- 达不到发布门槛时任务失败，不生成低质量或近乎空白的期刊。
- 每次运行生成结构化报告，记录成功来源、失败来源、候选数量、去重数量、最终数量和
  失败原因。
- 运行报告作为短期 GitHub Actions artifact 保存，不写入长期内容目录。
- 内容提交成功但 Pages 部署失败时，内容仍安全保留；手动重跑可只构建和部署现有内容。
- 手动补发使用相同日期计算、去重和渲染逻辑，避免形成第二套流程。

## 12. 安全与版权

- 工作流使用最小权限：内容写入、Pages 部署和身份令牌权限按 job 分配。
- 新闻链接只允许 `http` 和 `https` 协议。
- RSS 摘要先去除 HTML、脚本和内联样式，再按长度截断。
- 不执行来源返回的 HTML 或 JavaScript，不下载媒体附件。
- 不复制全文、付费内容、大段 feed description 或媒体图片。
- 每条内容必须显示来源并链接到原文。
- 商业化前应重新核对各来源的许可和使用条款。

## 13. 测试策略

### 13.1 单元测试

覆盖：

- 北京时间目标日期与 72 小时时间窗口。
- RSS/Atom 时间解析和时区转换。
- URL 规范化和跟踪参数移除。
- 标题清洗、相似标题和历史重复判断。
- 栏目配额、来源上限和补充内容选择。
- 发布门槛与幂等退出行为。

### 13.2 集成测试

使用仓库内固定 RSS/Atom 和 HN JSON 样本，不依赖实时网络。
测试完整流程从候选数据生成日期 Markdown、README 最近期刊区、HTML、归档和 RSS。

### 13.3 快照与构建检查

- 固定日期和固定输入应产生稳定输出。
- 对同一天重复生成两次，第二次不应产生文件差异。
- Markdown 和 HTML 中不得包含危险标签或非 HTTP(S) 链接。
- 归档、上一期/下一期、RSS 和静态资源链接必须有效。
- CI 在 pull request 和普通 push 上运行测试、代码风格检查和离线站点构建。

## 14. 验收标准

首版完成时必须满足：

1. 本地命令可以为指定日期生成一份完整日刊。
2. 日刊保存为按年月组织的一天一份 Markdown。
3. 目标日期新闻不足时，最多向前补充两天并正确标注。
4. 最近 30 天已发布链接和重复标题不会再次入选。
5. 达不到最低内容门槛时不会提交期刊。
6. 重复运行同一日期不会创建重复内容或无意义提交。
7. GitHub Actions 包含北京时间 09:00 主调度、09:20 备用调度和手动补发。
8. GitHub Pages 能展示最新一期、单期、月度归档、总归档、来源页和关于页。
9. RSS 可被常见阅读器解析。
10. 自动化不需要用户提供 AI 或新闻 API 密钥。
11. 全部测试和离线构建通过。

## 15. 后续扩展边界

首版稳定后，可以独立增加前端搜索、多来源事件聚类、GitHub Issues 投稿、邮件订阅或
可选的 AI 摘要。任何需要付费密钥或改变版权边界的扩展都必须显式启用，
不能改变首版默认的免费无密钥路径。
